# Production Architecture

## 1. 製品境界

SIMS-Blog-Managerは、候補抽出、Search Consoleデータ管理、改善履歴、効果測定を担当します。
SIMS-Coreは、改善依頼の解釈、記事改善、品質確認、フィードバック生成を担当します。

## 2. 実行フロー

1. Blog-Managerが標準のSIMS_REQUEST_TEXT_V2を生成（V1も後方互換）
2. 利用者が任意の対応AIへ依頼文を渡す
3. SIMS-Core実装が記事改善を実行
4. 人が読める改善成果を出力
5. 回答末尾にSIMS_FEEDBACK_V1 1.1を出力
6. Blog-Managerへ結果登録

## 3. 内部パイプライン

- Intake：識別子・数値・制約を抽出
- Intent：検索意図と改善目的を確定
- Audit：現状の強み・不足・変更範囲を判断
- Improve：タイトル、導入、見出し、FAQ、本文等を改善
- Internal Link：候補を評価し、テキストリンクをアフター本文へ埋め込み、ブログカード位置を指定
- Quality：検索意図、正確性、可読性、実装可能性を検査
- Feedback：本文と整合するJSONを生成

内部パイプラインは実装詳細であり、利用者へEngine番号を要求しません。

## 4. 設計原則

- 既存記事の良い部分を維持する
- データを捏造しない
- 一般論だけで終わらせない
- ビフォー・アフター＋理由で、コピペ可能な成果物を返す
- テキストリンクはアフター完成文へHTMLで埋め込む
- ブログカードは具体的な挿入位置と完全URLを示す
- 隠れた思考過程ではなく、短い改善根拠を示す
- JSONを本文の実施内容と一致させる
